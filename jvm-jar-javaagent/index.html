☕ 第一性原理解析

JVM、JAR 與 -javaagent
為什麼更新 dd-java-agent 後要重啟？

從 .jar 檔的本質開始，一步步理解 JVM 載入機制、javaagent hook、以及為什麼「換了檔案不會自動生效」

📦 最根本的問題：.jar 檔到底是什麼？

.jar（Java ARchive）其實就是一個 ZIP 壓縮檔，裡面裝的是編譯好的 Java class 檔案和一些設定檔。

📦
dd-java-agent.jar（本質上是 .zip） ├── META-INF/MANIFEST.MF（設定檔：入口點等） ├── datadog/trace/agent/TracerInstaller.class ├── datadog/trace/bootstrap/Agent.class ├── datadog/trace/instrumentation/... └── 數千個 .class 檔（編譯好的 Java 程式碼）
🎒 背包比喻

.jar 就像一個背包，裡面裝了很多便當盒（.class 檔案）。每個便當盒是一個編譯好的 Java 類別，JVM 需要的時候會從背包裡拿出來用。

重點是：背包是打包好的，JVM 啟動時會把它「打開」並把內容放到工作桌上（記憶體）。之後就算你偷偷換了背包，工作桌上的便當盒不會自動換。

⚙️ 跟 Python / Node.js 的關鍵差異

☕ Java（JVM）

編譯 + 一次性載入

原始碼 → 編譯成 .class → 打包成 .jar → JVM 啟動時 ClassLoader 載入到記憶體 → 之後不再讀磁碟

換了 jar → JVM 不知道 → 必須重啟

🐍 Python / Node.js

直譯 / 每次讀取

原始碼 → 直譯器每次 import / require 時讀取檔案 → 部分支援 hot reload

換了檔案 → 某些情境下不用重啟（但也有 cache）

這就是為什麼 Java 世界裡「更新 jar 後要重啟」是常識，而 Python 開發者可能覺得「換了檔案怎麼不會自動生效？」

🔌 -javaagent 是什麼？

當你用這個指令啟動應用程式：

java -javaagent:dd-java-agent.jar -jar myapp.jar

-javaagent 告訴 JVM：「在啟動應用程式之前，先載入這個特殊的 jar，讓它有機會修改其他 class 的 bytecode。」

🔧 安檢比喻

想像一棟大樓（你的應用程式）要開門營業。

-javaagent 就像在大樓門口安排了一個安檢員（dd-java-agent）。他在每個員工（class）進門之前，會在他們的工牌上貼一個追蹤貼紙（instrumentation）。

這樣每個員工做了什麼事，都會被記錄下來（trace span）。而且員工自己不知道被貼了貼紙——這就是為什麼叫「自動計裝」。

🔄 一句話總結

🎯 核心原理

JVM 的 ClassLoader 在啟動時一次性把 jar 裡的 class 載入到記憶體。之後 JVM 只用記憶體裡的版本，不再回去讀磁碟上的 jar。

所以更新 dd-java-agent.jar 的正確流程是：

1. 下載新版 jar → 2. 覆蓋舊檔案 → 3. 重啟 JVM（必須）

不重啟 = 記憶體裡還是舊版 = 新版形同虛設。

✅ 更新後不會變的東西

-javaagent:dd-java-agent.jar 參數 — 不用改

其他 JVM 啟動參數 — 不用改

應用程式的設定 — 不受影響
