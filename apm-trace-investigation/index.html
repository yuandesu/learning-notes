每一步的角色：

1. .NET Tracer（v3.40.0）— 在應用程式 process 內運行，產生 span，透過 Unix socket 送給同節點的 Agent

2. trace-agent（Agent 7.58.2 的一部分）— 跑在 K8s 每個節點上（DaemonSet），接收 span 後轉發到 Datadog Intake

3. Intake — Datadog 後端的接收端點，接收 span 資料

4. Backend — 處理、索引、儲存

5. UI — 你在 Datadog console 看到的 APM 畫面

🔑 關鍵認知

如果 trace 在 UI 上看不到，問題可能出在這 5 步的任何一步。不能直接假設是 Datadog 後端壞了——可能 span 根本沒送出去。

🔑 從架構圖看關鍵事實

1. trace-agent 和 log-agent 是同一個 Agent binary 裡的不同 process，走不同的 endpoint。一個壞了不影響另一個。

2. App Pod 跟 Agent Pod 透過 Unix socket 通訊（不是 TCP）。如果 Agent Pod 被驅逐或重啟，socket 會斷裂，span 就送不出去。

3. 每個 K8s 節點上只有 一個 Agent Pod（DaemonSet 排程）。如果這個節點的 Agent 壞了，該節點上所有 App Pod 的 trace 都會受影響。

🔀 關鍵前提：Log 和 Trace 走的是完全不同的路

這是理解整個事件的核心。客戶說「trace 消失但 log 正常」，這不矛盾——因為它們根本是兩條獨立的管線。

📝 Log Pipeline

App → Log Agent → agent-http-intake.logs.datadoghq.com

事件期間：完全正常

客戶提供的 CSV 有 100,000 筆 log，每筆都有有效的 128-bit @traceid

這證明 .NET Tracer 在正常運行（有產生 trace context），Log Agent 也在正常送資料

📡 Trace Pipeline

App → trace-agent → trace.agent.datadoghq.com

事件期間：完全斷裂

Span 從每天 114,000 降到只有 142 個（集中在第一小時），之後 22 小時為零

Span 從未到達 Intake——不是「到了但沒被索引」，而是根本沒送到

🍕 外送比喻

想像一家披薩店同時用兩家外送平台送餐：UberEats（Log）和 Foodpanda（Trace）。

某天 Foodpanda 的外送員罷工了，但 UberEats 照常送。客人在 Foodpanda 上看不到任何訂單，但 UberEats 的訂單全部正常送達。

這不代表「披薩店關門了」（Tracer 壞了）——披薩還是有在做（log 裡有 trace_id 證明）。問題出在 Foodpanda 的外送員（trace-agent）停止取餐了。

🔀 為什麼 Log 有 trace_id 不代表 Trace 有送出去？

這是客戶最容易搞混的地方。

@traceid 是 .NET Tracer 在應用程式記憶體裡產生的一個 ID。它會同時被注入到：

.NET Tracer
產生 trace_id
→
注入到 Log
跟著 Log 管線走
.NET Tracer
同一個 trace_id
→
送 Span 給 trace-agent
這一步斷了

所以 log 裡有 trace_id 只能證明 Tracer 有產生 trace context，不能證明 span 有成功送到 trace-agent。

📋 結論

🎯 根本原因

客戶 K8s 節點上的 trace-agent 停止轉發 span，持續約 23 小時。這是客戶端基礎設施的問題，不是 Datadog 後端的問題。

Span 從未到達 Intake（跟「到了但沒被索引」不同）。當新的 EC2 節點上線後，帶著全新的 Agent pod，trace 自行恢復。

為什麼 Log 正常但 Trace 斷了？

Log Agent
獨立 process
≠
trace-agent
獨立 process

它們是同一個 Datadog Agent binary 裡的不同 process，走不同的 pipeline、不同的 endpoint。一個壞了不影響另一個。

為什麼無法確認具體觸發原因？

事件當時的 K8s 節點已被回收（EKS auto-scaling），原始 agent log 不復存在。所有 flare 裡的節點都是事件結束後 7.5～48 小時才啟動的全新節點。

需要問客戶的問題

問題 1
2026-04-03 03:00 JST 前後，K8s cluster 上是否有節點擴縮事件、DaemonSet 更新、或 cluster 維護？
問題 2
K8s events 裡有沒有相關紀錄？（kubectl get events -n datadog）
問題 3
Agent DaemonSet 的 resource limits/requests 設定是什麼？有沒有 OOMKill 紀錄？
問題 4
當時是否正在進行 Helm 或 Datadog Operator 的升級？
