Datadog Java Tracer — SQL 擷取流程解析

說明 APM Trace 中的 SQL 查詢是從哪裡取得的，以及正規化的完整過程

SQL 是從哪裡取得的？

Datadog Java Tracer 取得的 SQL 並非來自資料庫端（Oracle、MySQL 等）的 log，而是在應用程式透過 JDBC 送出 SQL 的當下，由 tracer 在 JDBC 層進行攔截（hook）所取得。

🚫

不是從這裡取得

Oracle / MySQL 的 SQL log、slow query log、audit log

✅

而是從這裡取得

應用程式呼叫 Connection.prepareStatement(sql) 時，tracer 攔截並取得 SQL 字串

核心架構示意

/* 呼叫鏈示意 */

應用程式
  └─ Connection.prepareStatement("SELECT * FROM orders WHERE id = ?")
       │
       ├─ 【Datadog Tracer 攔截點】
       │   ByteBuddy hook → 取出 SQL 字串 → SQLNormalizer.normalize()
       │   → 存入 span 的 resource name
       │
       └─ Oracle / MySQL（收到的是帶 ? 的 prepared statement）
            └─ 執行 SQL，但 Datadog 不讀取 DB 端的 log
💡 因為攔截點在 JDBC 層，所以不論使用哪種資料庫（Oracle、MySQL、PostgreSQL 等），行為都一致。

SQL 擷取的完整流程（點擊各步驟查看詳細）

1
應用程式呼叫 prepareStatement()
JDBC API — 應用程式側

應用程式透過 JDBC 建立 PreparedStatement，SQL 字串在此時傳入。

應用程式側（客戶程式碼）
// 應用程式的典型寫法
String sql = "SELECT member_code, tsr_code FROM rismon WHERE id = ?";
PreparedStatement ps = connection.prepareStatement(sql);
ps.setInt(1, 12345);  // ← bound parameter（Datadog は取得しない）
ps.executeQuery();
2
ByteBuddy hook 攔截
AbstractConnectionInstrumentation.java

Datadog tracer 透過 ByteBuddy，在 JVM 啟動時對所有實作 Connection 介面的類別進行 bytecode 注入，攔截 prepare* 開頭的方法。

dd-java-agent/instrumentation/jdbc/src/main/java/datadog/trace/instrumentation/jdbc/AbstractConnectionInstrumentation.java
// 定義攔截哪些方法
transformer.applyAdvice(
    nameStartsWith("prepare")              // prepareStatement / prepareCall
        .and(takesArgument(0, String.class))  // 第一引數為 SQL 字串
        .and(returns(hasInterface(named("java.sql.PreparedStatement")))),
    ConnectionPrepareAdvice.class.getName());

// 方法結束時執行（OnMethodExit）
@Advice.OnMethodExit(suppress = Throwable.class)
public static void addDBInfo(
    @Advice.Argument(0) final String sql,         // ← SQL 字串在此取得
    @Advice.Return final PreparedStatement statement) {
  DBQueryInfo info = DBQueryInfo.ofPreparedStatement(sql); // 正規化
  contextStore.put(statement, info);                       // Statement 物件關聯
}
3
SQLNormalizer 正規化
SQLNormalizer.java / DBQueryInfo.java

取得的 SQL 字串會通過 SQLNormalizer.normalize() 進行正規化，將 literal 值替換成 ?，避免敏感資料進入 trace。

dd-java-agent/agent-bootstrap/.../jdbc/DBQueryInfo.java
// PreparedStatement → キャッシュあり
public static DBQueryInfo ofPreparedStatement(String sql) {
    return CACHED_PREPARED_STATEMENTS.computeIfAbsent(sql, NORMALIZE);
}

// 通常の Statement → キャッシュなし（毎回正規化）
public static DBQueryInfo ofStatement(String sql) {
    return NORMALIZE.apply(sql);
}

// 両方とも最終的にここを通る
public DBQueryInfo(String sql) {
    this.sql = SQLNormalizer.normalize(sql); // ← 正規化の本体
}
internal-api/.../normalize/SQLNormalizer.java
// 正規化の判定ロジック（3種類の literal を ? に置換）
if (isQuoted(utf8, sequenceStart, sequenceEnd)        // 'string' → ?
    || isNumericLiteralPrefix(utf8, sequenceStart)    // 123, -1.5 → ?
    || isHexLiteralPrefix(utf8, sequenceStart, sequenceEnd)) { // x'FF' → ?
  utf8[sequenceStart] = '?';
}
4
Span の resource name に設定
JDBCDecorator.java

正規化済みの SQL が span の resource name として設定され、Datadog APM UI に表示される。

dd-java-agent/instrumentation/jdbc/src/main/java/datadog/trace/instrumentation/jdbc/JDBCDecorator.java
private AgentSpan withQueryInfo(AgentSpan span, DBQueryInfo info, ...) {
  if (null != info) {
    span.setResourceName(info.getSql());      // ← 正規化済み SQL を span に設定
    span.setTag(DB_OPERATION, info.getOperation());
  }
  return span.setTag(Tags.COMPONENT, component);
}

APM UI の Span List に表示される SQL はこの resource name です。

5
Trace Agent 側でさらに難読化
datadog-agent（サーバー側）

tracer 側の正規化は「ベストエフォート」。最終的な難読化は Datadog Agent（trace-agent）が行う。

internal-api/.../normalize/SQLNormalizer.java — Javadoc
/**
 * This is not an obfuscator, and the strings produced by this class
 * must be passed through obfuscation in the trace agent.
 *
 * （この class は obfuscator ではなく、ここで生成された文字列は
 *   trace agent 側でさらに難読化処理が必要）
 */
Tracer（アプリ側）と Trace Agent（サイドカー）の二段階で処理されることで、機密データの保護を強化している。

⚠️ Bound Parameter は絶対に取得されない
setString(1, "secret")、setInt(1, 99) などで渡した値は、 正規化どころか tracer に渡ることすらない。JDBC の内部メカニズムでバインドされるため、 SQL 文字列自体には含まれない。

實際案例對照

案例 A：PreparedStatement（最常見）

應用程式程式碼 
// 應用程式傳入的 SQL
String sql = "SELECT MEMBER_CODE, TSR_CODE,"
           + " SETTLINGDAY, AMOUNT_UNIT"
           + " FROM rismon"
           + " WHERE id = ? AND status = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setInt(1, 12345);      // 不被 tracer 取得
ps.setString(2, "active"); // 不被 tracer 取得
APM Trace 顯示的 SQL 
// ? 維持不變（原本就是 placeholder）
SELECT MEMBER_CODE, TSR_CODE,
  SETTLINGDAY, AMOUNT_UNIT
FROM rismon
WHERE id = ? AND status = ?

案例 B：一般 Statement（含 literal 值）

應用程式程式碼 
// 直接把值嵌入 SQL（不推薦，但 tracer 仍支援）
String sql = "SELECT * FROM orders"
           + " WHERE year = 2024"
           + " AND status = 'PAID'";
Statement stmt = conn.createStatement();
stmt.execute(sql);
APM Trace 顯示的 SQL 
// SQLNormalizer が literal を ? に置換
SELECT * FROM orders
WHERE year = ?        ← 2024 が置換された
AND status = ?       ← 'PAID' が置換された

案例 C：IN 子句（多個值）

應用程式程式碼 
String sql = "SELECT id FROM products"
           + " WHERE category IN (1, 2, 3)";
stmt.execute(sql);
APM Trace 顯示的 SQL 
// 各數字がそれぞれ ? に置換される
SELECT id FROM products
WHERE category IN (?, ?, ?)

📌 客戶の認識は正しい
「DB のログからではなく、アプリが JDBC に渡した SQL を、フックした場所で拾っている」という認識は正確です。 加えて、tracer は SQL の構造（正規化済み）のみを取得し、実際のパラメータ値は一切取得しません。
