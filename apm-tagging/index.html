🏷️ 第一性原理解析

Datadog APM Tagging
Tag 是什麼？加在哪？怎麼加？

客戶問了兩個問題：①怎麼加 tag ②加在什麼上面。其實這是同一個問題的兩面。

🏷️ 最根本的問題：Tag 到底是什麼？

在 Datadog 裡，Tag 就是一個 key:value 的標籤。例如 env:production、service:payment-api、region:ap-northeast-1。

它的作用是讓你能 filter（篩選）、aggregate（聚合）、scope（限定範圍） 你的監控資料。

🏪 便利商店比喻

Tag 就像商品上的標籤：「飲料」「冷藏」「促銷中」「7-11限定」。

你可以用這些標籤快速找到東西（filter）、統計各類別銷量（aggregate）、只看冷藏區（scope）。

但重點是：標籤貼在「商品」上，不是貼在「貨架」上。同理，Tag 貼在「telemetry 資料」上，不是貼在「伺服器」上。

🏷️ 客戶的困惑點

客戶問：「対象となるリソースは何か（フロントのコード、サーバー、など）」

這個問法暗示客戶以為 tag 是貼在「伺服器」或「前端代碼」這種物理/邏輯元件上的。

但實際上，tag 附著在 telemetry 資料的層級上。APM 的資料是分層的：

🟧 Host（主機：EC2 / VM / 物理伺服器）
🔷 Container / Pod（容器 / K8s Pod）
🟪 Service（服務：payment-api, user-service...）
　 Trace（一次完整的請求旅程）
🟩 Span（單個處理：HTTP call / DB query / 函式呼叫）

每一層都可以有 tag，而且上層的 tag 會自動繼承到下層。這就是為什麼不能簡單說「tag 加在伺服器上」——它可以加在 5 個不同的層級上，效果各不相同。

📍 Q②：Tag 附著在哪一層？

點擊每一層展開詳細說明。從最外層（Host）到最內層（Span），tag 的粒度越來越細。

上層的 tag 會向下繼承：Host tag → 所有 Container → 所有 Service → 所有 Span

1
Host（主機）
最廣範圍
▸

VM、EC2 instance、物理伺服器等。Datadog Agent 跑在這一層。

Host tag 會自動繼承到該主機上所有的 metrics、traces、logs。是影響範圍最廣的 tag。

Tag 範例

region:ap-northeast-1
role:web
env:prod
instance-type:m5.xlarge

繼承行為

Host 上所有 container 產生的所有 span 都會自動帶上這些 tag。你不需要在每個服務裡重複設定。

│ 繼承 ↓
2
Container / Pod / Serverless 函式
容器粒度
▸

K8s Pod、Docker container、ECS Task、Lambda function 等。

從 Pod label、Docker label、ECS task definition 等自動抽取的 tag，會附著在該容器產生的 trace span 上。

Tag 範例

kube_namespace:production
kube_deployment:payment-api
app:checkout
team:backend

Unified Service Tagging

K8s 環境下，Pod label
tags.datadoghq.com/env
tags.datadoghq.com/service
tags.datadoghq.com/version
會被 Admission Controller 自動轉換成環境變數。

│ 繼承 ↓
3
Service（APM 服務）
⭐ Datadog 推薦起點
▸

一個「服務」= 一個應用程式或微服務（payment-api、user-service、order-batch 等）。

env、service、version 這三個是 Datadog 的標準 tag（Unified Service Tagging），是 APM 裡最重要的 tag。它們把 metrics、traces、logs 串連在一起。

三大標準 Tag

env:production — 環境
service:payment-api — 服務名
version:2.1.0 — 版本號

為什麼最重要？

這三個 tag 是 APM Service Map、Service Page、Trace Metrics 的基本軸。沒有它們，APM 的大部分功能無法正確運作。

│ 繼承 ↓
4
Span（處理單元）
最細粒度
▸

一個 Span = 一次 HTTP request、一次 DB query、一次函式呼叫等。是 APM 裡最小的資料單位。

開發者可以在程式碼裡用 Tracer API 手動加 tag，附著在特定的 span 上。這是最靈活但也需要改程式碼的方式。

Tag 範例

customer_id:12345
feature_flag:new-checkout
error.type:TimeoutException
http.url:/api/v1/payment

用途

在 Trace Explorer 裡 filter、建立自訂的 trace metrics、設定精確的 monitor alert。只影響加了 tag 的那個 span。

── 以下非 APM ──
5
Frontend（RUM — 非 APM）
另一個產品
▸

前端（ブラウザ / モバイル）的監控不是 APM，而是 RUM（Real User Monitoring）。

但透過在 RUM SDK 初始化時設定 env、service、version，可以與 APM trace 相互關聯和篩選。

設定方式不同

RUM 的 tag 透過 SDK 初始化參數設定，不是 Agent 或 Tracer API。方式與 APM 完全不同。

與 APM 的關聯

設定相同的 env / version 後，可以在 RUM 的 session 裡直接跳轉到對應的 APM trace，反向也可以。

⚙️ Q①：每一層怎麼設定 Tag？

知道了 tag 附著在哪一層之後，「怎麼加」就自然對應上了。點擊展開每層的具體設定方式。

1
Host Tag 的設定方法
Agent 設定
▸

在 Datadog Agent 的設定檔或環境變數中設定：

方法 A：設定檔

datadog.yaml 裡的 tags: 項目

tags:
- region:ap-northeast-1
- role:web

方法 B：環境變數

DD_TAGS="region:ap-northeast-1,role:web"

容器環境（Docker / K8s）推薦用這個方式。

此外，AWS / GCP / Azure 等雲端整合的 tag 會自動繼承，不需要手動設定。

2
Container / Pod Tag 的設定方法
K8s Label / ECS Task Def
▸

依環境而異：

Kubernetes

在 Deployment / Pod 的 metadata 裡加 label：

tags.datadoghq.com/env: production
tags.datadoghq.com/service: payment-api
tags.datadoghq.com/version: 2.1.0

Datadog Admission Controller 會自動轉換成 DD_ENV 等環境變數。

ECS / Docker

在 ECS Task Definition 裡設定環境變數：

DD_ENV=production
DD_SERVICE=payment-api
DD_VERSION=2.1.0

Docker 環境也可以用 container label。

3
Service Tag 的設定方法
⭐ 環境變數（推薦）
▸

Datadog 最推薦的做法：Unified Service Tagging

環境變數（推薦）

在服務啟動時設定三個環境變數：

DD_ENV=production
DD_SERVICE=payment-api
DD_VERSION=2.1.0

Tracer 會自動讀取並附加到所有 trace 上。

非容器環境

直接在執行命令中設定：

DD_ENV=prod DD_SERVICE=my-svc DD_VERSION=1.0 /bin/my-service

或寫在 systemd 的 service 設定檔裡。

4
Span Tag 的設定方法
程式碼 Tracer API
▸

在應用程式碼裡，用各語言的 Datadog Tracer API：

Go

span.SetTag("customer_id", "12345")

Python

span.set_tag("customer_id", "12345")

Java

span.setTag("customer_id", "12345")

Ruby / Node / etc.

各語言都有類似的 set_tag 或 setAttribute API。

這是唯一需要改程式碼的方式，但也是粒度最細的——可以對每個 span 個別加不同的 tag。

5
Frontend（RUM）的設定方法
RUM SDK 初始化
▸

前端不用 APM Tracer，用 RUM SDK：

SDK 初始化

datadogRum.init({
service: 'web-app',
env: 'production',
version: '3.0.0',
...
})

注意

這不是 APM 的 tag，而是 RUM 的 tag。但設定相同的 env / service / version 可以讓 RUM 與 APM 資料互相關聯。

📊 完整對照：5 層 × Tag 來源 × 設定方法 × 範圍

層級 Tag 附著在 設定方法 影響範圍 例
1 Host 主機（VM / EC2） DD_TAGS / datadog.yaml / 雲端整合自動 該 host 上所有 telemetry region:ap-northeast-1
2 Container Pod / Container / Task K8s label / ECS task def / Docker label 該容器的 trace span kube_namespace:prod
3 Service APM 服務 DD_ENV / DD_SERVICE / DD_VERSION 該服務的 trace + trace metrics service:payment-api
4 Span 單個處理單元 程式碼 span.set_tag() 只影響該 span customer_id:12345
5 RUM 前端 session / event RUM SDK 初始化 init() RUM 資料（非 APM） version:3.0.0

🔑 給客戶的回答 — 一句話版

Q① 怎麼加 Tag？

依層級不同：Host 用 Agent 設定、Container 用 K8s label、Service 用環境變數（DD_ENV 等，推薦起點）、Span 用程式碼 set_tag()、Frontend 用 RUM SDK。

Q② 加在什麼上面？

不是「伺服器」或「前端代碼」這種分法。Tag 附著在 5 層 telemetry 資料上：Host → Container → Service → Span → (RUM)。上層 tag 會自動繼承到下層。
