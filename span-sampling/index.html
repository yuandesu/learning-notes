🎯 第一性原理解析

Datadog Span Sampling
完全理解指南

用一個電商下單的真實 trace 範例，走過 Ingestion → Retention 的所有 sampling 規則

🛒 先搞懂：一個 Trace 長什麼樣子？

用一個電商網站的「使用者下單」請求來看。使用者點了「結帳」按鈕，觸發了以下的呼叫鏈：

Trace（一次完整的下單請求）
│
├─ Span 1: GET /checkout           ← root span（= head）  [web-frontend]    200ms
│   ├─ Span 2: POST /api/order     ← 子 span             [order-service]    180ms
│   │   ├─ Span 3: SELECT * FROM products   ← DB query    [postgres]          15ms
│   │   ├─ Span 4: POST /payment/charge     ← 外部呼叫    [payment-service]   80ms
│   │   │   └─ Span 5: Stripe API call      ← 第三方      [stripe]            60ms
│   │   └─ Span 6: INSERT INTO orders       ← DB write    [postgres]          10ms
│   └─ Span 7: SendEmail           ← 非同步              [notification-svc]   5ms

這個 trace 有 7 個 span，橫跨 5 個 service。Sampling 的決定會影響這 7 個 span 是全部保留、全部丟棄、還是只保留其中幾個。

🛒 Sampling 的兩個階段

階段一：Ingestion（攝取）

Span 要不要送進 Datadog？

在 Tracer + Agent 端決定

5 種規則：Head-based、User-defined、Error、Rare、Single Span

階段二：Retention（保留）

已經進來的 span 要保留多久？

在 Datadog Backend 端決定

2 種規則：Intelligent Retention、Custom Filters

接下來的章節會用上面的電商範例，逐一走過每個規則。

🎲 Head-based Sampling（預設）

「Head」指的是 trace 的第一個 span（root span）。Sampling 的決定在這個「頭」產生的瞬間就做了——要嘛整個 trace 保留，要嘛整個 trace 丟掉。

Head-based（Datadog 預設）

在 trace 的第一個 span 產生時就決定

決定傳遞到所有下游 service

同一個 trace 的 span 要嘛全留、要嘛全丟

✅ 輕量、即時、不需暫存

Tail-based（參考對比）

等 trace 的所有 span 都收集完才決定

可以根據完整 trace 資訊做判斷

例如「整個 trace 有 error 才保留」

❌ 需暫存所有 span，資源消耗大

🎲 Agent 和 Tracer 的分工

Head-based sampling 不是單一元件的決定，而是 Agent 和 Tracer 的協作：

Step 1 — Agent 計算 sampling rate
Agent 觀察流量
→
web-frontend
500 req/s → rate 2%
→
admin-service
1 req/s → rate 100%
Agent 根據每個 service 的流量算出比例。目標：整體控制在每秒 10 個 trace（DD_APM_TARGET_TPS=10）。高流量降低、低流量保持 100%。
Step 2 — Agent 把 rate 傳給 Tracer
Agent
「web-frontend
你用 2%」
→
Tracer
收到 rate: 2%
Tracer 定期向 Agent 拿最新的 sampling rate。Agent 像教練告訴選手跑多快，但實際跑的是 Tracer。
Step 3 — Tracer 在 root span 產生時做決定
Tracer 擲骰子
2% 機率
→
中了！留下
sampling-priority: 1
→
HTTP header 傳遞
→ 所有下游 service
使用者發起 GET /checkout，web-frontend 的 Tracer 擲骰子。中了 → 在 HTTP header（x-datadog-sampling-priority: 1）標記「留」。下游的 order-service、payment-service、postgres 看到 header 就知道這個 trace 要保留，不會重新擲骰子。
Step 4 — Agent 轉發到 Intake
Tracer 送出
7 個 span
→
Agent
照送到 Intake
→
Intake 收到 ✓
Tracer 決定「留」的 span，Agent 原封不動轉發到 Datadog Intake。Agent 在這裡的角色是搬運工，不會再做二次篩選。

🎲 User-defined Sampling Rules（Tracer 層級）

覆蓋 Agent 的自動 rate，針對特定 service 或 resource 設定自己的比例：

DD_TRACE_SAMPLING_RULES='[
  {"service": "payment-service", "sample_rate": 1.0},
  {"service": "health-check",   "sample_rate": 0.01},
  {"name": "GET /api/v1/users", "sample_rate": 0.5}
]'
🛒 電商場景

payment-service 處理金流，每一筆交易都不能漏 → 100%。health-check 被 load balancer 每秒打幾百次，沒必要全留 → 1%。Ingestion reason 標記為 rule。

🛡️ 被 Head-based 丟掉的 trace，還有三道安全網

Head-based sampling 擲骰子沒中的 trace 不一定就消失了。Agent 還有三個額外的 sampler 會撈回重要的 span：

3
Error Sampler
Agent 層
▸

對沒被 head-based sampling 抓到的 error trace，Agent 額外保留。

# datadog.yaml
apm_config:
  error_traces_per_second: 10   # 預設每秒 10 個
# 或 DD_APM_ERROR_TPS=10
🛒 電商場景

上面的下單請求裡，Span 4: POST /payment/charge 回傳了 500 error（Stripe 扣款失敗）。但這個 trace 剛好沒被 head-based 的 2% 選中。Error sampler 會額外把它撈回來，因為 error trace 對 debug 很重要。

Ingestion reason：error。設成 0 = 停用，設成很大的數字 = 保留所有 error trace。

4
Rare Sampler
Agent 層
▸

對低流量的 service/resource/operation 組合，Agent 額外取樣。

# datadog.yaml
apm_config:
  enable_rare_sampler: true     # 預設開啟
# 或 DD_APM_ENABLE_RARE_SAMPLER=true
🛒 電商場景

你的系統裡有一個 POST /admin/reset-cache endpoint，一天只被呼叫 3 次。Head-based sampling 每秒取 10 個 trace，但這個 endpoint 一天只有 3 個 trace，很可能全部被 drop。Rare sampler 確保這種低流量組合至少保留一些樣本。

Ingestion reason：rare。

5
Single Span Sampling
Span 層級
▸

即使整個 trace 被 drop，仍保留特定的單一 span。

DD_SPAN_SAMPLING_RULES='[
  {"service": "payment-service", "sample_rate": 1.0, "max_per_second": 100}
]'
🛒 電商場景

你建了一個 span-based metric 監控 payment-service 每分鐘的交易金額（用 @transaction_amount 做 SUM）。Head-based 只取 2%，metric 會不準。Single span sampling 確保每一個 payment-service 的 span 都被保留（每秒最多 100 個），即使完整 trace 被 drop 了。

注意：這些 span 沒有完整 trace context，Trace Explorer 裡看不到完整 flame graph。Ingestion reason：single_span。

🌐 Remote Configuration 是什麼？

讓你從 Datadog UI 直接修改 sampling rate，不需要改程式碼、不需要改環境變數、不需要重新部署應用程式、不需要重啟 Agent。

🌐 它控制的是誰？

Remote Configuration 可以控制兩個層級，指令都透過 Agent 中繼傳遞：

控制 Agent 層（不需重啟 Agent）

可以遠端修改的 Agent 設定：

- DD_APM_TARGET_TPS（head-based sampling 的每秒目標 trace 數）

- DD_APM_ERROR_TPS（error sampler 的每秒上限）

- DD_APM_ENABLE_RARE_SAMPLER（rare sampler 開關）

🛒 場景

月底快超過 ingestion 預算。從 UI 把 DD_APM_TARGET_TPS 從 10 降到 5，幾十秒內生效，不用碰伺服器。

控制 Tracer 層（不需重新部署應用程式）

可以遠端修改的 Tracer 設定：

- Per-service sampling rate（例如 payment-service 100%）

- Per-resource sampling rate（例如 GET /checkout 50%、GET /health 1%）

- Adaptive sampling（設定月度 ingestion 預算，Datadog 自動調整 rate）

🛒 場景

部署了新版本，想暫時 100% 取樣看有沒有 bug。從 UI 設 payment-service 100%，觀察 1 小時，沒問題後改回 10%。全程不需要碰伺服器。

即使本機設了 DD_APM_TARGET_TPS=10，從 UI 改成 20 之後，20 會生效。Remote Config 永遠贏。
原理：為什麼不需要重啟？

Agent 本來就會定期跟 Datadog Backend 通訊（送 metrics、traces、logs）。Remote Configuration 利用這條已經存在的連線，反過來讓 Backend 把設定指令「推」回給 Agent。

具體運作流程

1. Agent 定期 polling — Agent 每隔幾秒向 Datadog Backend 的 Remote Configuration endpoint 發請求：「有沒有新的設定給我？」

2. 你在 UI 改了設定 — 例如把 payment-service 的 sampling rate 從 10% 改成 100%。這個變更存在 Datadog Backend。

3. Agent 下一次 polling 拿到新設定 — Backend 回覆：「有，payment-service 的 rate 改成 100% 了。」

4. Agent 套用並傳遞 — 如果是 Agent 層的設定（如 TARGET_TPS），Agent 直接套用。如果是 Tracer 層的設定（如 per-service rate），Agent 在下一次 Tracer 來要 sampling rate 時，把新的 rate 傳給 Tracer。

為什麼不需要重啟？

因為 sampling rate 在 Agent 和 Tracer 裡就是記憶體裡的一個可變變數，隨時可以被覆寫。不像 JVM 的 ClassLoader 一次性載入就鎖死。

❌ JVM + jar（之前講的）

Class 被 ClassLoader 載入到記憶體後，不再讀磁碟

換了 jar → JVM 不知道 → 必須重啟

像出車前拿到的紙條，要換路線得回車行

✅ Sampling rate

Rate 是記憶體裡的一個數字，隨時可以改

Agent polling 拿到新值 → 直接覆寫 → 即時生效

像無線電，老闆隨時可以喊你改路線

🚕 計程車比喻

datadog.yaml = 出車前老闆給你的紙條：「今天跑新宿路線」。出車後紙條鎖在抽屜裡，要換路線得回車行（= 重啟 Agent）。

Remote Configuration = 老闆用無線電喊你：「改跑澀谷路線」。你正在開車就能馬上轉向（= 不用重啟）。

無線電（polling 連線）本來就一直開著，因為你本來就要用它回報位置（送 telemetry）。Remote Configuration 只是反過來也用這條線傳指令。

🌐 前提條件

- Agent 版本 7.41.1 以上

- Agent 啟用 Remote Configuration（DD_REMOTE_CONFIGURATION_ENABLED=true）

- Datadog 帳號有 APM Remote Configuration Write 權限

- Tracer library 版本符合要求（各語言不同）

📦 階段二：Retention（已經進來的 span 保留多久？）

Ingestion 決定 span 要不要送進 Datadog。Retention 決定送進來的 span 保留多久、哪些可以在 Trace Explorer 裡被搜尋到。

6
Intelligent Retention Filter
自動 · 免費
▸

Datadog 自動保留兩類資料，不需要任何設定：

Diversity sampling：每 15 分鐘，對每個 env + service + operation + resource 組合，至少保留 1 個 trace。保留 30 天。

1% flat sampling：對所有 ingested span 均勻取 1%。保留 30 天。

🛒 電商場景

GET /checkout 每天 50 萬次，POST /admin/reset-cache 每天 3 次。Diversity sampling 確保兩者都有樣本可看。不計入 indexed span 費用。

7
Custom Retention Filters
使用者自定義
▸

在 Datadog UI 的 Retention Filters 頁面建立，用 tag/attribute query 指定保留哪些 span，保留 15 天。

Filter 名稱Query電商場景
高金額交易@transaction_amount:>100金額超過 100 的交易全部留下做 audit
Prod 錯誤env:prod status:error所有 production error 都要能事後查
慢結帳resource_name:"GET /checkout" @duration:>2s結帳超過 2 秒的要能 debug
VIP 客戶@customer_id:VIP-001VIP 客戶的每一筆請求都要追蹤
Stripe 失敗service:payment-service @http.status_code:5*金流失敗全部保留

分兩種模式：

- Span-level：只保留符合 query 的那些 span

- Trace-level：只要 trace 裡有任何 span 符合，整個 trace 都保留（可看完整 flame graph）

🔑 一句話總結

🎯 核心概念

Agent 算 rate、Tracer 做決定、Agent 轉發。Head-based sampling 在 root span 產生的瞬間就決定整個 trace 的命運。被 drop 的 trace 還有 Error / Rare / Single Span 三道安全網。進了 Datadog 之後，Intelligent Retention 自動保留多樣性樣本，Custom Filters 保留你關心的特定 span。

Remote Configuration 讓你從 UI 直接改 Agent 和 Tracer 兩個層級的設定，不需要碰伺服器，優先順序最高。
