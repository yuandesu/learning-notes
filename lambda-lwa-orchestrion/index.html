🧱 第一性原理解析

Lambda Web Adapter + Orchestrion
為什麼 Trace 送不出去？

從最基礎的概念開始，一步步理解客戶的問題

🧱 先搞懂：普通 Web Server 怎麼運作？

你寫了一個 Go Web API。正常情況下，它是這樣跑的：

普通 Web Server 運作方式
Go Web API
:8080
← 持續監聽 HTTP 請求 →
使用者

你的程式碼就是一個 持續運行的 HTTP Server，用 net/http.ListenAndServe(":8080", ...) 在那邊等請求進來。它「活著」直到你關掉它。

🧱 Lambda 的運作模式完全不同

Lambda 不是 Web Server。它是一個「事件處理函式」：

Lambda 運作方式
使用者
→
API Gateway
→
Lambda Runtime
JSON event 進
JSON response 出

Lambda 函式接收的是 JSON event（不是 HTTP 請求），回傳的也是 JSON response。它不監聽任何 port，每次執行完就「凍結」（freeze），等下一個 event。

🍔 漢堡店比喻

普通 Web Server = 一間持續營業的漢堡店，廚師一直站在那裡等客人點餐。

Lambda = 一個外送廚師，每次收到一張訂單（JSON event），做完一個漢堡就「暫停」，等下一張訂單才「醒來」。他沒有自己的店面（沒有 port）。

🧱 Lambda 會自動設定一個關鍵環境變數

每個 Lambda 函式啟動時，AWS 會自動注入：

LAMBDA_FUNCTION_NAME=my-function

這個環境變數是 Lambda 運行環境的「身分證」。很多工具（包含 Datadog tracer）會用它來判斷：「我是不是跑在 Lambda 裡面？」

記住這個環境變數，它是後面所有問題的核心。

🔌 問題：我有一個現成的 Web API，不想改寫成 Lambda 格式

你有一個用 Go 寫的 Web API，它用 net/http 監聽 :8080。你想把它跑在 Lambda 上（省成本、自動擴展），但你不想把程式碼改成 Lambda handler 格式。

這就是 AWS Lambda Web Adapter（LWA）的用途。

🔌 LWA 是什麼？一個「翻譯員」

🌐 翻譯員比喻

你是一個只會說中文的廚師（Go Web API），但客人說的是日文（Lambda JSON event）。

LWA 是一個翻譯員，坐在你旁邊：客人用日文點餐 → 翻譯成中文給你 → 你做好漢堡 → 翻譯員把你的中文回覆翻回日文給客人。

你完全不需要學日文（不需要改程式碼）。

技術上怎麼做到的？

LWA 是一個用 Rust 寫的輕量 process，以 Lambda Extension 形式運行。它在 Lambda 容器內：

1. 接收 Lambda Runtime 傳來的 JSON event
2. 轉換成 HTTP 請求，送給你的 Go Web API（localhost:8080）
3. 接收 HTTP response，轉換回 JSON 送回 Lambda Runtime

你的 Go Web API 完全不知道自己跑在 Lambda 上。它以為自己還是在監聽 :8080 的普通 Server。

🔌 但是！環境變數還在

雖然你的 App「以為」自己是普通 Web Server，但 Lambda 環境裡的那個環境變數依然存在：

LAMBDA_FUNCTION_NAME=my-function ← 還在！

這就是衝突的根源。App 以為自己不在 Lambda 上，但環境變數告訴所有工具「你在 Lambda 裡」。

🎻 問題：Go 沒辦法 runtime 自動注入 tracing

Java / Python / Node.js

有 runtime（JVM / 直譯器），可以在程式「跑起來之後」偷偷注入 tracing 程式碼。

-javaagent:dd-java-agent.jar
ddtrace-run python app.py

✅ 不需要改程式碼

Go

編譯成原生二進位檔（binary），跑起來就是機器碼，沒有地方可以 hook。

傳統做法：手動在程式碼裡加
import "gopkg.in/DataDog/dd-trace-go.v1"

❌ 必須改程式碼

🎻 Orchestrion 的解法：在編譯的時候動手腳

📝 作文比喻

你寫了一篇作文（Go 原始碼），要交給老師批改（Go 編譯器）。

Orchestrion 是一個「作文助理」，在你交出去之前，自動在適當的地方加上注釋和標記（tracing 程式碼），然後才交給老師。老師看到的是「已經加好標記的作文」。

你不需要自己加標記，助理幫你搞定。

Orchestrion 編譯流程
你的 Go 原始碼
main.go
→
🎻 Orchestrion
AST 分析 + 注入
dd-trace-go
→
Go 編譯器
go build
→
已計裝的 Binary
內含 tracing

Orchestrion 用 Go 工具鏈的 -toolexec 機制，在編譯時攔截原始碼，用 AST（抽象語法樹）分析，自動插入 dd-trace-go 的計裝程式碼。編譯出來的 binary 已經內建 tracing 功能。

結果：你完全不需要改任何一行程式碼，但產出的 binary 會自動產生 Datadog APM trace。

🎻 Orchestrion 注入的 dd-trace-go 會做什麼判斷？

這是關鍵！dd-trace-go（Datadog 的 Go tracer library）啟動時，會檢查環境變數：

LAMBDA_FUNCTION_NAME 存在嗎？

不存在 → 普通模式

Tracer 自己把 trace 透過 localhost:8126 送給 Datadog Agent，或直接送到 Datadog backend。

✅ Trace 正常送出

存在 → Lambda 模式

Tracer 切換行為：把 trace 交給 Datadog Lambda Extension 來處理，依賴 Extension 在 Lambda 生命週期結束時 flush。

⚠️ 需要 Extension 配合

💥 三個東西疊在一起，問題就來了

現在把三個概念組合起來：

1. Lambda 自動設定 LAMBDA_FUNCTION_NAME

2. LWA 讓 Go App「以為」自己是普通 Web Server

3. Orchestrion 自動注入的 dd-trace-go 看到 LAMBDA_FUNCTION_NAME → 切換 Lambda 模式

但 LWA 的運作方式與標準 Lambda invocation 不同——App 是以持續運行的 Web Server 模式在跑，不是標準的「一個 event → 一次 invocation → freeze」流程。Tracer 期待的 Lambda Extension flush 機制在這個混合架構下無法正確觸發。

結果：Trace 產生了，但永遠送不到 Datadog APM。

💥 一句話總結根因

🎯 根本原因

dd-trace-go 看到 LAMBDA_FUNCTION_NAME → 以為自己在標準 Lambda 裡 → 用 Lambda 模式送 trace。

但 App 實際上是透過 LWA 當作普通 Web Server 跑的 → Lambda 模式的 flush 機制不適用 → Trace 卡住。

身分認知錯亂：App 以為自己是 Web Server，但 Tracer 以為自己在 Lambda 裡。

✅ 客戶的回避方法：刪掉那個環境變數

客戶在 App 啟動時呼叫：

os.Unsetenv("LAMBDA_FUNCTION_NAME")
刪掉環境變數後
dd-trace-go 初始化
→
LAMBDA_FUNCTION_NAME
不存在 ✓
→
普通模式 ✓
自己送 trace

修改前

LAMBDA_FUNCTION_NAME=my-func

→ Tracer 進入 Lambda 模式
→ 等待 Extension flush
→ Trace 遺失

修改後

LAMBDA_FUNCTION_NAME

→ Tracer 進入普通模式
→ 自己送 trace 到 Datadog
→ Trace 正常顯示 ✓

本質上就是欺騙 tracer：讓它以為自己不在 Lambda 裡，這樣它就會用普通模式自己把 trace 送出去。

💭 客戶想知道的：有沒有「更正式」的解法？

客戶目前的 os.Unsetenv 方法可行，但他覺得這是 workaround，想知道有沒有官方推薦的做法。

這就是你需要問 TEE 的核心問題：

🔍 需要確認的方向

1. dd-trace-go 是否有設定可以強制指定 tracer 模式（不依賴環境變數自動偵測）？

2. Datadog Lambda Extension 是否能與 LWA 架構正確配合？如果能，需要什麼額外設定？

3. Orchestrion 是否有 LWA 專用的設定或已知的最佳實踐？

4. os.Unsetenv("LAMBDA_FUNCTION_NAME") 是否為目前的官方建議做法？
